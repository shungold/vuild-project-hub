Step 1: 構造分析

チャンネルの全メッセージ + 全スレッド返信を読み込み、案件の構成を分析する。ここで判定したパターンが以降の全Stepに影響する。

このステップで適用されるルール 案件名ルール（チャンネル名をそのまま使用）/ スキャン対象範囲（本文+全スレッド）/ チーム抽出（users.info API、VUILDメンバー vs 外注先を区別）

5つの判定軸 → パターン

判定軸選択肢発動するルール
furniture数単数 / 複数B 複合furniture構成の整理ルール
エリア構成単一 / 複数フロアB エリア × furnitureマッピング
建築工事の混在なし / ありC furniture vs 建築の振り分け
WSイベントなし / ありD WSイベント分析
年次/関連案件単発 / 複数年E 別案件分離 + related_projects

パターンA: シンプル — 追加処理なし

A furniture少数・単一エリア・建築なし・WSなし。ストレートにStep 2へ。

パターンB: 複数エリア

B エリア × furnitureマッピングを作成。以降の全Step（写真分類・specs・ピン配置）で参照。

エリアマッピングルール

各エリア（フロア/ゾーン）にどのfurnitureが配置されるか明示的に整理。ユーザー確認必須。階層/タイプ/連数で分割。

既存furniture同一判定: furniture.jsonを検索 → 「既存と同一か？」をユーザーに確認。

CLAUDE.md: エリアマッピング / 複合furniture / 既存furniture同一判定

パターンC: 建築工事混在

C VUILDが製作・移動可能 → furniture / 壁床天井固定・工務店施工 → 案件ページ。

パターンD: WSイベントあり

D WSイベントを検出し、参加者・日付・対象furnitureを分析。WS写真 vs 製作ナレッジ vs 竣工写真を区別。

WS写真ルール

参加者が作業 = WS写真（各イベントmax 5枚）。職人の製作作業 = productionナレッジ。ファイル名だけでWS判定しない。

CLAUDE.md: WSイベント管理

パターンE: 複数年/関連案件

E 常に別案件として登録。年次タブは使わない。related_projects で紐づけ（relation: ws/year/phase/related）。

確認ゲート 構造分析レポートをユーザーに提示。確認・補足・修正を受けてから次へ進む。

2-6. 写真選定（3フェーズ）

Phase A: テキストシグナルで一次分類

シグナル優先度

メッセージテキスト（最高）> スレッド親（高）> 時系列クラスタ（高）> 前後5件（中）> 投稿者パターン（中）> タイムスタンプ（中）

バッチ投稿（3枚以上・テキストなし）: 全写真をPhase Bへ。1枚ずつ個別に判定する。

Phase B: 画像認識で二次分類

対象 & ルール

各グループ上位10枚 + 判定不能 + バッチ写真。Step 1のfurniture特徴と照合。

B エリアマップ: 主役（面積50%以上 or 中央）→ OK / 背景 → NG / 複数furniture全景 → 案件のみ

D WS写真: 参加者作業 → WSタブ / 職人作業 → productionナレッジ / ファイル名で判定しない

ViewCapture hero判定: 3Dパース → OK / CAMパス・2D図面・板取り図 → NG

Phase C: スコアリング & ユーザー確認

ルール内容
スコアリング横長 +10 / 黒部 +15 / リアクション数 / 画像サイズ
枚数上限案件: max 7枚 / furniture: max 7枚 / WS: max 5枚/イベント
案件ページ空間を見渡せる引き画のみ。ディテールNG。各エリアから最低1枚（B）
hero_photosRhinoアクソメ > ViewCapture（3D） > 空欄。竣工写真はheroに使わない
サムネイル大型什器 → gallery_photos[0] / 中小什器 → hero_photos[0]
圧縮max 1600px / JPEG q80 / ≤1600px & <500KBならスキップ

Step 3: クロスチェック

全データの振り分け後、各furnitureの「部屋」を開けて中身を俯瞰検証する。1件ずつの振り分けでは見逃す「混入・矛盾・欠落」をここで検出。

4つのチェック

チェック1: 混入
この部屋の中身は全て正しいfurnitureのものか？
例: 7Fキッチンの部屋に6Fの塗装情報 → 6Fに移動
チェック2: 整合性
specs・ナレッジ・写真・コストが互いに矛盾していないか？
例: specsは「シナ合板」、ナレッジは「ラワン」 → ユーザーに確認
チェック3: 欠落
あるべき情報が拾えていないfurnitureはないか？
例: 写真はあるのにナレッジ/コストがゼロ → 案件の部屋を再確認
チェック4: 案件俯瞰
案件全体のバランスは取れているか？コストは整合するか？
例: furniture合計 ≠ 案件全体コスト → 振り分け漏れの可能性
