## 1. 背景と目的

Project Hubは現在、BAUMパターン（1チャンネル = 1 furniture）のスキャンロジックしか持っていない。
実際のVUILD案件は多様なパターンがあり、以下の課題がある：

- **楽天パターン**: 1チャンネルに複数年度 × 複数furnitureが混在
- **富山パターン**: 建築内装と置き式什器が混在
- **尾道パターン**: 1チャンネルに複数furniture（内装なし）

さらに、Slackに蓄積された製作ナレッジ（加工・塗装・組立の知見）が断片的なメモのまま散逸しており、
案件完了後に「どうやって作ったか」を振り返れない。

### v3.0 のゴール

1. **多パターン対応のスキャンロジック** — どんなチャンネル構造でもWEBに正確に整理できる
2. **製作ドキュメント** — furnitureごとに加工〜塗装〜搬送の手順書を自動生成
3. **案件ドキュメント** — 案件ごとに体制・現場条件・教訓をまとめた振り返り資料を自動生成
4. **差分スキャン** — 案件が増えても軽量にスキャンできる仕組み

## 3. 新機能：製作ドキュメント

### 3.1 概要

案件進行中にSlackから自動収集したナレッジ（断片的なメモ）を、案件完了時にfurnitureごとの「製作手順書」として自動整理する。

```
案件進行中                          案件完了時
┌─────────────────┐              ┌──────────────────────┐
│ Slackから自動収集    │              │ furnitureページに      │
│ ナレッジとして蓄積    │    ──→     │ 「製作ドキュメント」タブ  │
│ (断片的なメモ)       │   自動整理   │ (まとまった手順書)      │
└─────────────────┘              └──────────────────────┘
```

### 3.2 セクション構成

| セクション | 内容 | 収集キーワード |
|-----------|------|-------------|
| **素材** | 合板種類、厚み、芯材 | シナ, ラワン, MDF, 合板, mm |
| **加工** | CNC設定、取り数、注意点 | ShopBot, 切削, CAM, 加工データ |
| **組立** | 接合方法、金物、手順 | ビス, ボンド, JCB, 組立, 接合 |
| **下地・塗装** | 研磨→下地→塗装の全工程 | サンダー, キシラデ, ウレタン, 研磨, 塗装 |
| **搬送** | 梱包方法、車種、養生 | 搬入, 搬出, 梱包, JITBOX, トラック |
| **トラブル・教訓** | 失敗と対策 | 割れ, 欠け, 不具合, 変更, 注意 |

### 3.3 実例：尾道 客室ベンチA

```
素材:
  座面: ラワン合板（ルーバー構成）
  脚: スギ無垢材 30mm

加工:
  ShopBot切削
  注意: 角部の欠けが発生しやすい → バッファ材を見込むこと

組立:
  竜骨+脚: ビス斜め打ち接合
  補強: Lアングルで竜骨-脚間を補強（梁材の割れ対策）
  接合金物: cabineo8 M6

下地・塗装:
  1. キシラデクリア塗装（ガン吹き）
  2. 研磨
  3. キシラデクリア 2回目
  4. ウレタン塗装（仕上げ）
  工数目安: 見積10人工 → 実績40人工（大幅超過に注意）

搬送:
  JITBOX + 2tゲート車
  ベルト10〜15本で固定、ウレタン端材で緩衝
  脚裏にカグスベール貼り（現場で）

トラブル・教訓:
  - Aタイプのミラー形状8台が不足 → 図面段階で左右ミラーの数量確認を徹底
  - 梁材に割れ発生 → スギ無垢は含水率チェック、Lアングル補強を標準化
```

### 3.4 データ構造

```javascript
// knowledge.json に追加
"production_documents": [
  {
    "id": "PD001",
    "furniture_id": "onomichi-bench-a",
    "status": "draft | published",
    "sections": [
      { "key": "material",  "title": "素材",       "content": "..." },
      { "key": "machining", "title": "加工",       "content": "..." },
      { "key": "assembly",  "title": "組立",       "content": "..." },
      { "key": "finishing",  "title": "下地・塗装", "content": "..." },
      { "key": "shipping",  "title": "搬送",       "content": "..." },
      { "key": "lessons",   "title": "トラブル・教訓", "content": "..." }
    ],
    "source_knowledge_ids": ["K020", "K021"],
    "generated_at": "2025-03-17",
    "reviewed_by": null
  }

```

### 3.5 運用フロー

1. **進行中** → Slackスキャンでナレッジ自動収集（既存の仕組み）
2. **案件完了時** → 「ドキュメント生成して」コマンドで、ナレッジをテンプレートに自動整理
3. **レビュー** → draft状態で生成 → 担当者が確認 → published
4. **WEB表示** → furnitureページに「製作ドキュメント」タブで表示

## 4. 新機能：案件ドキュメント

### 4.1 概要

furnitureの製作ドキュメントとは視点が異なり、**「この案件をもう一回やるなら何を知っておくべきか」**をまとめる。

### 4.2 セクション構成

| セクション | 内容 |
|-----------|------|
| **クライアント・体制** | 関係者・窓口・工務店・外注先 |
| **現場条件** | 場所、搬入経路、車種、制約 |
| **設計判断** | 仕様変更の経緯と理由 |

| **外注・協力** | 外注先リスト・金額・連絡先 |

## 5. スキャンロジック v2

### 5.1 スキャン3モード

| モード | トリガー | 動作 |
|--------|---------|------|
| **差分スキャン** | 「Slackスキャンして」 | 登録済みチャンネルの新着のみ読む（軽量） |
| **初回スキャン** | 「初回スキャン {チャンネルID}」 | 新チャンネルの全量読み→構造分析→質問→データ生成 |
| **アイテム検索** | 「{アイテム名}をスキャンして」 | Slack横断検索→チャンネル特定→照合→差分提案 |

### 5.2 差分スキャン

```javascript
// data/scan_state.json で前回スキャン位置を管理
{
  "channels": {
    "C05AYV4KG3E": {
      "name": "#393_楽天optimism",
      "last_scanned_ts": "1775314516.102389",
      "last_scanned_at": "2026-04-05T10:00:00"
    }
  }
}
```

- Slack APIの `oldest` パラメータで前回以降の新着のみ取得
- `status: "completed"` かつ30日以上スキャンなし → 自動スキップ
- 完了案件も明示指定すれば読める

### 5.3 初回スキャンフロー

```
Step 1: チャンネル全量読み → 構造分析
  - 案件の種別判定
  - furniture / 建築工事 の識別
  - 年度分割の検出

Step 2: 質問フェーズ
  - 「furniture 4アイテム検出しました。この構成でOK？」
  - ユーザー確認を得てからデータ生成

Step 3: データ生成
  - projects.json / furniture.json / materials.json / knowledge.json 更新
  - 製作ドキュメント・案件ドキュメントをdraftで生成
  - 写真特定（黒部さん優先）→ DLリスト提示
  - scan_state.json にチャンネル登録
  - HTMLページ生成 → commit & push

Step 4: レビュー
  - WEB URLを提示して確認依頼
```

### 5.4 アイテム検索スキャンフロー

```
Step 1: Slack横断検索（slack_search_public）
  - 「タウンユニット」を含むメッセージ・チャンネルを特定

Step 2: チャンネル一覧表示
  - 既にスキャン済み / 未登録 を区別して表示
  - 未登録チャンネルがあれば初回スキャンを提案

Step 3: 情報照合
  - 各チャンネルからアイテムに関する情報を抽出
  - 仕様・数量・素材・金物・ナレッジを収集

Step 4: WEB照合・差分検出
  - 現在のfurniture.jsonと比較
  - 不足データ（重量null、hardware_ids空など）を検出
  - 新規ナレッジ候補を提示

Step 5: 更新提案
  - 差分を一覧表示してユーザーに確認
  - OKをもらったらWEBに反映
```

### 5.5 振り分けルール

#### furniture vs 建築工事

| 判定基準 | furniture | 建築工事 |
|---------|-----------|---------|
| 移動できるか | 搬入・搬出する | 壁/床/天井に固定 |
| VUILDが製作したか | 海老名工場で加工 | 工務店・設備屋が施工 |
| 他案件で再利用可能か | レンタル・転用できる | その建物専用 |

**重要ルール**: VUILDが家具として製作したものは全てfurniture。キッチンも置き式ならfurniture。

#### 金物の振り分け

- 金物は親アイテムに従う（キッチンの金物→キッチンfurniture、配管の金物→建築工事）
- 同じ金物が複数furnitureに使われる場合は両方に登録

#### 自動分類キーワード

| カテゴリ | キーワード | 行き先 |
|---------|-----------|--------|
| furniture | ShopBot, 加工, 組立, 搬入, 搬出, 納品, 塗装, 研磨, 梱包, 海老名, レンタル | furnitureページ |
| 建築工事 | 配管, 配線, 電気屋, 工務店, 給排水, 設備, 換気扇, 止水栓, 壁, 天井, 基礎 | 案件ページ |
| 金物 | 品番あり, monotaro, sugatsune, URL付き | 親アイテムに従う |
| ナレッジ | トラブル, 割れ, 不具合, 変更, 注意 | production knowledge |

#### コスト3階層

| 階層 | 行き先 | 判定キーワード |
|------|--------|-------------|
| クライアントへの請求額・内訳 | 案件コストタブ | 見積, 請求書, 内訳, 税込, 支払い |
| furniture単価・材料費 | furnitureコスト欄 | 1台あたり, 単価, 取り数, 合板○枚 |
| 外注費・協力費 | 案件コストタブ | 工務店, 電気屋, 配管, 運送, ○○さん手間 |
| コスト交渉・VE | 案件ナレッジ | 予算, 変更, 削減, VE, 浮いた分 |

## 9. 技術メモ
