# VUILDlibrary スキャンフロー完全ガイド

> **ルール原本**: `G:\マイドライブ\Claude_Projects\VUILDlibrary\CLAUDE.md`
> **最終更新**: 2026-04-07
> **このドキュメントは参照用まとめ。ルール変更はCLAUDE.mdで行うこと。**

---

## 概要

Slackチャンネルから案件情報を収集し、VUILD Project Hub（GitHub Pages）に掲載するまでの全フロー。
3つのスキャンモードがあり、初回スキャンが最も複雑。

| モード | トリガー | 内容 |
|--------|---------|------|
| **初回スキャン** | `初回スキャン {チャンネルID}` | 新チャンネル全量読み → 5ステップで完全処理 |
| **差分スキャン** | `Slackスキャンして` | 登録済みチャンネルの新着のみ |
| **アイテム検索** | `{アイテム名}をスキャンして` | Slack横断検索 → 照合 → 差分提案 |

---

## 初回スキャンの5ステップ

```
Step 1: 構造分析 → ★ユーザー確認
Step 2: データ生成（振り分け）
Step 3: クロスチェック → ★ユーザー確認 → 完了判定+ドキュメント生成
Step 4: ページ生成 → デプロイ
Step 5: WEB確認依頼
```

★ = ユーザー確認ゲート（ここで止まり、確認後に次へ進む）

## Step 1: 構造分析

チャンネル全メッセージを読み、案件の構成を分析する。
ここで判定したパターンが、以降の全Stepに影響する。

### 1-1. 5つの判定軸

全メッセージから以下を同時にチェック:

| 判定軸 | 選択肢 | 例 |
|--------|--------|-----|
| furniture数 | 単数 or 複数 | 1個のベンチ vs キッチン+スツール+芝キューブ |
| エリア構成 | 単一 or 複数（フロア/ゾーン） | 1フロア vs 6F+7F |
| 建築工事の混在 | なし or あり | 什器のみ vs 什器+防水工事+植栽 |
| WSイベント | なし or あり | なし vs セルフビルドWS、塗装体験 |
| 年次/関連案件 | 単発 or 複数年/関連あり | 単発 vs 楽天2023+2024 |

### 1-2. パターン別の追加処理

判定結果に応じて、以下の追加処理が発動する:

**A. シンプル案件**（furniture少・単一エリア・建築なし・WSなし・単発）
→ 追加処理なし。ストレートにStep 2へ

**B. 複数furniture × 複数エリア**
→ **エリア × furnitureマッピング** を作成（以降の全Stepで参照）

| 適用ルール | 内容 |
|-----------|------|
| エリアマッピング | 各エリアにどのfurnitureが配置されるか明示的に整理 |
| 複合furniture構成の整理 | 階層/タイプ/連数で分割、ID命名規則に従う |
| 既存furniture同一判定 | furniture.jsonを検索し「既存と同一か？」をユーザー確認 |

**C. 建築工事混在**
→ **furniture vs 建築の振り分け判定** が追加

| 適用ルール | 内容 |
|-----------|------|
| 情報の振り分けルール | VUILDが製作・移動可能 → furniture / 壁床天井固定 → 案件 |

**D. WSイベントあり**
→ **WSイベント分析** が追加

| 適用ルール | 内容 |
|-----------|------|
| WSの検出と分析 | 各WSの名前・日付・対象furniture・参加者を特定 |
| WS写真の判別 | 参加者作業=WS / 職人作業=production / 竣工使用=WS以外 / ファイル名だけで判定しない |
| WS写真の枚数 | 各WSイベントごとに max 5枚 |

**E. 複数年/関連案件**
→ **別案件として分離** + `related_projects` で紐づけ

| 適用ルール | 内容 |
|-----------|------|
| 年次案件ルール | **年次タブは使わない。** 常に別案件 |
| 関連案件の整理ルール | relation: "ws" / "year" / "phase" / "related" |

### 1-3. チーム体制の分析

| 適用ルール | 内容 |
|-----------|------|
| 名前取得 | `users.info` APIでUID → real_name（推測しない） |
| VUILDメンバー vs 外注先 | 「依頼」「見積受領」「手配」→ 外注先。team[]にはVUILDメンバーのみ |
| メンバー選定基準 | 投稿数上位（全投稿の5%以上目安） |
| 他案件チームのコピー禁止 | チャンネル参加者の投稿内容から抽出 |

### 1-4. 構造分析レポート → ★ユーザー確認

以下をレポートとして提示し、**確認・補足・修正を受けてから次へ進む**:
- 案件名・クライアント・所在地
- パターン判定結果
- エリア × furnitureマッピング（パターンB）
- furniture一覧（★既存 or 新規の判定）
- 建築工事一覧（パターンC）
- WSイベント一覧（パターンD）
- 関連案件の分割方針（パターンE）
- 写真投稿者と枚数
- チーム体制

## Step 2: データ生成

ユーザー確認済みの構造に基づき、1件ずつメッセージ/写真/ファイルを正しい場所に振り分ける。
**Step 1で確定したパターンとエリアマップを常に参照しながら実行。**

### 2-1. ナレッジ収集 & 分類

| 適用ルール | 内容 |
|-----------|------|
| Slackスタンプマッピング | 📝→progress / ⚠️→trouble / 💡→howto / 🚩→spec |
| ナレッジ自動分類 | スタンプなしでもAIがキーワード2つ以上一致で候補判定。source: "ai_auto" |
| 除外条件 | 30文字未満、bot/join/leaveメッセージはスキップ |
| scope振り分け | production（他案件で再利用可能）→ furnitureページ / design（案件固有）→ 案件ページ |
| エリアマップ照合（パターンB） | 複数エリアの情報が混在するため、どのエリアの話か文脈から特定 |

### 2-2. 金物候補の検出 & 登録

| 適用ルール | 内容 |
|-----------|------|
| ECリンク検出 | monotaro/amazon/sugatsune等のURL → スクレイピング → candidates登録 |
| テキスト名検出 | 金物名キーワード → URL/品番なしで候補登録 + プレースホルダー |
| furniture紐づけ | メッセージ文脈 + 親スレッドテーマで推定。不明→案件レベル |
| スタンプ処理 | ✅→hardwareCatalog / 🔍→pending / ❌→dismissed / 💰→ordered |
| 既存データ保護 | 登録済みのURL/価格/品番を上書きしない |
| 重複チェック | product_urlでドメイン+パス照合 |

### 2-3. furniture specs 深掘り

| 適用ルール | 内容 |
|-----------|------|
| 情報源の優先順 | 1.Slackメッセージ → 2.見積PDF → 3.加工依頼書・図面 → 4.ナレッジ |
| 記述精度基準 | material: 合板種類+厚み+部位（「合板」だけNG） / finish: 部位別塗装仕様 / note: 外注先・担当 |
| dimensions | W×D×H フォーマット。情報源: Slack → PDF → 図面 → 3Dモデル |
| エリアマップ照合（パターンB） | 塗装情報のエリア特定が必須（桜葉ステイン=7F、ウレタン=6Fなど） |

### 2-4. コスト抽出 & 時系列分析

| 適用ルール | 内容 |
|-----------|------|
| 自動抽出 | Slackの金額パターン検出 → furniture名共起で分類 |
| 手動優先 | 既存の手動入力値は上書きしない |
| 時系列分析 | コスト関連メッセージ+PDFを時系列で並べ → 発信元分類 → 変遷追跡 → 最新/最終値をcost_breakdownに |
| 値の型 | 数値（確定値）と文字列（"約9万円"）の両方許容 |
| 振り分け | furniture単体コスト → cost_breakdown / 案件全体 → cost_overview |
| コスト表示の優先ルール | production_cost（実績）> cost_breakdown（見積ベース）> 「コスト未取得」 |

### 2-5. ドキュメント収集 & 分類

| 適用ルール | 内容 |
|-----------|------|
| type分類 | estimate/invoice → 見積タブ / drawing/fabrication → 図面タブ / reference/contract/schedule → ドキュメントタブ |
| 振り分け優先順 | furniture名含む→そのfurniture / 部品名→該当furniture / 案件全体語→案件 / 判定不能→案件にデフォルト |
| 両方に関連 | 案件とfurniture両方に登録 |
| 見積PDF分析 | DL → Read で金額・内訳抽出 → furniture別振り分け |

### 2-6. 写真選定（3フェーズ）

**Phase A: テキストシグナルで一次分類**

| 適用ルール | 内容 |
|-----------|------|
| シグナル優先度 | メッセージテキスト(最高) > スレッド親(高) > 時系列クラスタ(高) > 前後5件(中) > 投稿者パターン(中) > タイムスタンプ(中) |
| バッチ投稿 | 1投稿3枚以上でテキストシグナルなし → 全写真をPhase B送り。**1枚ずつ個別判定** |
| スレッド返信 | 写真はスレッド返信にも多数含まれる。必ず含めること |

**Phase B: 画像認識で二次分類**

| 適用ルール | 内容 |
|-----------|------|
| 対象 | Phase A上位候補(各グループ上位10枚)+判定不能+バッチ写真 |
| 判定方法 | Step 1のfurniture特徴（形状・素材・色・サイズ）と照合 |
| エリアマップ照合（パターンB） | 主役判定: 面積50%以上 or 中央 → OK / 背景→NG / エリア全景→案件のみ |
| WS写真判別（パターンD） | 参加者作業→WSタブ / 職人作業→productionナレッジ / ファイル名で判定しない |
| ViewCapture hero適格判定 | 3Dパース→OK / CAMパス・2D図面・板取り図→NG |

**Phase C: グルーピング + スコアリング + ★ユーザー確認**

| 適用ルール | 内容 |
|-----------|------|
| スコアリング | 横長+10 / 黒部+15 / リアクション数 / 画像サイズ |
| 写真優先度 | 黒部竣工写真 > 完成状態 > 使用シーン。横長優先 |
| 枚数上限 | 案件: max 7枚 / furniture: max 7枚 / WSイベント: max 5枚 |
| 案件ページ固有 | 空間を見渡せる引き画のみ。ディテールNG→furnitureへ。エリア別に最低1枚ずつ |
| hero_photos | Rhinoアクソメ > ViewCapture(3Dパース) > 空欄（竣工写真はheroに使わない） |
| サイズ別サムネイル | 大型什器→gallery_photos[0] / 中小什器→hero_photos[0] |
| 写真不足時 | 竣工→完成に近い施工中→縦長→ViewCapture（加工途中/工場/養生は不可） |
| 圧縮ルール | max 1600px / JPEG quality 80 / スキップ: ≤1600px & <500KB |

### 2-7. 3Dモデル収集

| 適用ルール | 内容 |
|-----------|------|
| 検出 | .3dm / .obj / .stl をチャンネル本文+スレッドから収集 |
| 選定 | furniture名で紐づけ → 最新版のみ。「スタディ」「tmp」「検討」は除外 |
| WEB | model_url に登録 → DLリンク表示 |

### 2-8. チーム抽出

| 適用ルール | 内容 |
|-----------|------|
| 名前取得 | `users.info` APIでreal_name |
| VUILDメンバー vs 外注先 | 外注先はteam[]に入れない。ドキュメント/コスト欄で言及 |
| 選定基準 | 投稿数上位（全投稿の5%以上目安） |

## Step 3: クロスチェック（部屋別の俯瞰検証）

Step 2で各furniture/案件に振り分けたデータを、**furniture単位・案件単位で俯瞰して検証**する。
1件ずつの振り分け判定では見逃す「混入・矛盾・欠落」をここで検出する。

### チェック1: 混入チェック

各furnitureの「部屋」の中身を並べて、**場違いなもの**を探す。

| チェック対象 | 確認内容 | 混入の例 |
|------------|---------|---------|
| ナレッジ | そのfurnitureの話として通るか | 7Fキッチンの部屋に6Fの塗装情報 |
| 写真 | 全て同じfurnitureが写っているか | 別furnitureが主役の写真が紛れている |
| 金物 | 構造に対して金物リストが妥当か | ベンチの部屋にキッチン水栓金具 |
| コスト | 金額感がspecs/サイズと整合するか | スツールに100万円（案件全体の見積が紛れている） |

→ 混入検出時: 正しいfurnitureへの移動を提案

### チェック2: 整合性チェック

同じfurniture内で情報同士を突き合わせ、**矛盾**がないか確認する。

| 突き合わせ | 確認内容 | 矛盾の例 |
|-----------|---------|---------|
| specs × ナレッジ | 素材・塗装の記述が一致するか | specsは「シナ合板」、ナレッジは「ラワン合板」 |
| specs × 写真 | 写真の見た目とspecsが一致するか | 写真は白い仕上げ、specsは「桜葉ステイン（茶系）」 |
| コスト × ドキュメント | cost_breakdownと見積PDFが一致するか | cost_breakdown ¥250万、PDF ¥180万 |
| ナレッジのscope | production/designの分類が正しいか | 案件のdesignに「キャスターはロック付き」（→productionの方が適切） |

→ 矛盾検出時: 時系列で後のものが最終仕様の可能性を検討 + ユーザーに確認

### チェック3: 欠落チェック

各furnitureの情報充足度を一覧化し、**不自然な穴**を見つける。

```
            | specs | 写真 | ナレッジ | 金物 | コスト | 3Dモデル |
furniture A | OK    | OK   | OK      | OK  | OK    | OK      |
furniture B | 矛盾  | OK   | 少ない   | OK  | なし   | なし     |
furniture C | なし   | なし  | なし     | なし | なし   | なし     |
```

| パターン | 原因の推定 | 対応 |
|---------|----------|------|
| 全てなし | 別furnitureと一緒に語られ、個別振り分けできなかった | 再確認 or ユーザーに聞く |
| 写真あり、他なし | 情報が案件レベルに落ちている | 案件の部屋を再確認 |
| 特定カテゴリだけ欠落 | シンプルな製品で正常な場合も | ユーザーに提示 |

### チェック4: 案件レベルの俯瞰

| チェック内容 | 確認事項 |
|------------|---------|
| 写真バランス | 特定furnitureに偏っていないか、各エリアから選ばれているか |
| コスト整合 | furniture合計 vs 案件全体コスト → 差額が説明できるか |
| 案件ページの残留物 | designナレッジにproductionに移すべきものがないか |

### ★ユーザー確認

チェック結果を以下の形式で提示:
- **混入（自動修正提案）**: 移動先を提案 → OKで移動
- **矛盾（確認が必要）**: 情報Aと情報Bのどちらが最終仕様か確認
- **欠落（情報不足）**: 正常か異常かをユーザーに確認
- **scope修正提案**: production/designの変更を提案

### 完了判定 + ドキュメント生成（クロスチェック後に実行）

| 適用ルール | 内容 |
|-----------|------|
| 完了判定基準 | 案件: status="completed" / furniture: 紐づく全案件がcompleted |
| completed時 | 製作ドキュメント + 案件ドキュメントを必ず生成。各セクションの充足度チェック → 不足にプレースホルダー |
| active時 | ドキュメント生成スキップ（ナレッジ蓄積のみ） |
| production_documents | 素材/加工/組立/下地・塗装/搬送/トラブル・教訓 |
| project_documents | 体制/現場条件/設計判断/コスト/施工計画/WS/外注/教訓 |
| ステータス | draft → published（担当者レビュー後） |

**クロスチェックで修正済みのデータを使ってドキュメントを生成するため、このタイミングで実行する。**

## 差分スキャン（「Slackスキャンして」）

| 項目 | 内容 |
|------|------|
| 対象 | scan_state.jsonのactiveチャンネルのみ |
| 取得方法 | Slack APIの `oldest` パラメータで `last_scanned_ts` 以降の新着のみ |
| 自動スキップ | completed かつ 30日以上スキャンなし → スキップ（明示指定すれば読める） |
| 処理 | 5.1の振り分けルールで分類 → 差分をユーザーに提示 → 確認後反映 |
| 金物処理 | 🔩スレッド返信もチェック / 既存データ保護 / 重複チェック |
| 完了案件 | 新規ナレッジ追加 → ドキュメント再生成 / プレースホルダー埋め判定 |
| 最後 | scan_state.json更新 → git push |

---

## アイテム検索スキャン（「{アイテム名}をスキャンして」）

1. `slack_search_public` でアイテム名を横断検索
2. ヒットチャンネルを スキャン済み / 未登録 に区別して表示
3. 未登録チャンネル → 初回スキャンに移行するか確認
4. 登録済みチャンネルから情報照合 → furniture.jsonとの差分検出
5. 差分を一覧表示してユーザーに確認 → OKでWEB反映

### 写真の振り分け

| 被写体 | 行き先 | 上限 |
|--------|--------|------|
| 空間全体（引き画） | 案件ページ hero/gallery | max 7枚 |
| furnitureが主役 | furnitureページ gallery | max 7枚 |
| ディテール寄り | furnitureページのみ（案件NG） | galleryに含む |
| 建築工事が主役 | 案件 or 載せない | - |
| 施工中・加工中 | 載せない（WS例外あり） | - |
